# TTS notes

the voice synthesis problem is harder than it looks. a robotic voice in an emotionally sensitive context actively damages trust. if the AI responds to "I just found out I have cancer" in a flat corporate voice, the conversation is over.

---

quality spectrum:

old rule-based/concatenative systems (festival etc): intelligible, robotic. fine for navigation apps, wrong here.

neural TTS (current sota): near-human naturalness. wavenet, tacotron 2, VITS. the gap from 5 years ago is enormous.

main providers today:
- openai TTS: voices (alloy, echo, fable, onyx, nova, shimmer). high naturalness, ~300ms to first audio chunk. SSML support.
- elevenlabs: voice cloning from 1-minute sample. emotion-expressive synthesis (you can specify sadness, warmth, urgency). fine-grained control.
- google cloud TTS: WaveNet and Neural2 voice families. 380+ voices, 50+ languages. good SSML.
- azure TTS: expressive styles per voice — "empathetic", "excited", "sad", "calm". sub-200ms to first byte. good multilingual.

for prototype: openai TTS (easiest). for high-empathy production: elevenlabs with custom voice.

---

SSML for prosody control. this is how you make the voice feel appropriately warm without it sounding fake:

slow + low pitch = serious, attentive, warm
breaks (silence) = giving space, not rushing
soft volume = intimacy

example:
```
<speak>
  <prosody rate="slow" pitch="-2st">That sounds incredibly hard.</prosody>
  <break time="500ms"/>
  <prosody rate="medium" volume="soft">I'm here with you.</prosody>
</speak>
```

key insight: the SSML can be generated by the LLM alongside the response text. no separate prosody prediction module needed. just tell the model to output SSML markup along with the text and it can calibrate based on the emotional context it already understands.

---

prosody by conversation phase (rough guide):
- greeting: warm, medium pace, rising inflection
- listening/validation: slow, soft, lower pitch
- asking a question: rising intonation, pause before
- closing: warm, slightly slower, falling contour

these should change dynamically based on the affect state of the user — if they're distressed, the AI's voice should be slower and softer. if they're engaged and positive, the voice can be more energetic.

---

latency matters here too. for real-time conversation:
- LLM generates response text (streaming)
- TTS synthesis begins on first sentence (don't wait for full response)
- audio streams to client while TTS continues generating
- barge-in: user can interrupt mid-speech (VAD on output channel)

target: first audio chunk to user <1.5 seconds after they finish speaking.

---

voice persona design:
- not too young (low authority) 
- not too authoritative/clinical (intimidating)
- not too cheerful (tone-deaf in grief contexts)
- not too gendered
- consistent across sessions — users form a relationship with the voice

this is more of a product/UX decision than a technical one but it matters enormously.
